# GEMINI.md

## Часть 1. Общие правила разработки

### 1. Главная архитектура
`BM-concierge-backend` — отдельный проект ИИ-консьержа.
Он НЕ является частью CRM.

Правильная схема:
```text
Гость / Telegram / VK / сайт-чат
        ↓
BM-concierge-backend
        ↓
AI / база знаний / диалоги / заявки
        ↓
Supabase
        ↓
локальная CRM в Териберке забирает заявки
```

- CRM напрямую не подключать.
- SQLite-базу CRM не трогать.
- Код CRM не менять из этого проекта.

### 2. Что можно делать
Можно:
* развивать Console;
* улучшать UI;
* добавлять AI-провайдеров;
* улучшать базу знаний;
* добавлять локальные диалоги;
* добавлять локальные заявки;
* писать события/логи;
* тестировать mock/OpenRouter/DeepSeek;
* готовить Telegram/VK/Web Chat только отдельными согласованными этапами.

### 3. Что нельзя делать без подтверждения
Нельзя без отдельного подтверждения пользователя:
* менять схему Supabase;
* выполнять SQL `ALTER TABLE`, `DROP`, `DELETE`, `TRUNCATE`;
* менять RLS/policies/triggers/functions в Supabase;
* менять существующую таблицу `leads`, от которой зависит CRM;
* удалять локальные данные;
* чистить SQLite;
* подключать Telegram/VK/Web Chat;
* подключать CRM напрямую;
* добавлять большие зависимости;
* менять архитектуру проекта;
* автоматически подтверждать брони;
* менять шахматку/бронирования CRM;
* создавать договоры;
* проводить оплаты.

### 4. Supabase safety rules
Supabase уже используется локальной CRM.
Правило: БМ Консьерж не должен ломать текущую интеграцию Supabase → CRM.

Разрешено:
* проверять подключение;
* читать статус;
* создавать тестовые заявки только в dev-таблицу, если она явно указана;
* показывать ошибки;
* предлагать SQL-миграции текстом.

Запрещено:
* автоматически менять таблицу `leads`;
* менять поля, которые использует CRM;
* выполнять миграции без подтверждения;
* удалять записи;
* чистить таблицы.

Для разработки по умолчанию использовать:
```env
SUPABASE_LEADS_TABLE=leads_dev
```
Если `leads_dev` не существует — не создавать автоматически. Сначала показать, какие поля нужны, и запросить подтверждение.
В боевую таблицу `leads` писать только после явного подтверждения.

### 5. Секреты и ключи
Никогда не логировать и не возвращать в UI:
* `SUPABASE_SERVICE_ROLE_KEY`;
* `OPENROUTER_API_KEY`;
* `DEEPSEEK_API_KEY`;
* `CONSOLE_ACCESS_TOKEN`;
* любые env secrets;
* полные headers запросов.

В UI показывать только:
* `configured: true/false`;
* статус подключения;
* последнюю ошибку без секретов.

`.env`, `data/`, `data/console.db`, `data/dev-settings.json` должны быть в `.gitignore`.

### 6. Console
Console должна оставаться самостоятельной рабочей панелью.
Обязательные разделы:
* Главная
* Диалоги
* Тест чата
* Заявки
* База знаний
* Неразрешённые
* Подключения
* Supabase / CRM
* Автономность
* События / Логи
* Настройки

Редизайн не должен удалять функции.

### 7. База знаний и ответы AI
ИИ-консьерж должен отвечать только по базе знаний.

Запрещено:
* придумывать цены;
* придумывать наличие номеров;
* придумывать скидки;
* подтверждать бронь;
* обещать вид из окна, если это не указано;
* использовать WhatsApp;
* давать Telegram администратора;
* публиковать приватные контакты;
* выдумывать услуги;
* обещать северное сияние;
* обещать трансфер;
* обещать актуальную дорогу/погоду.

Если точной информации нет:
* не писать «в базе знаний нет»;
* не раскрывать внутреннюю механику RAG;
* мягко направить к администратору.

Телефон администратора:
```text
+79318022151
```
WhatsApp не использовать.

### 8. Lead logic
Создание заявки должно определяться нашей логикой `LeadExtractor`, а не решением LLM.

Если гость спрашивает:
* наличие на даты;
* цену;
* хочет забронировать;
* просит связаться;
* указывает даты/количество гостей;
* хочет изменить/отменить бронь,
создаётся локальная заявка в SQLite.

Заявка сначала сохраняется локально, потом пробует отправиться в Supabase.
Если Supabase недоступен:
* заявка не теряется;
* статус `error`;
* ошибка сохраняется без секретов;
* можно повторить отправку.

### 9. Проверки после изменений
После backend/TS-изменений запускать:
```bash
npm run build
```
Для UI-правок можно ограничиться ручной проверкой.
Перед тяжёлыми изменениями один раз выполнить `git status --short`, если git доступен.
Не запускать лишние тяжёлые проверки без необходимости.

### 10. Обязательные тесты поведения
После изменений проверять:
1. «Можно ли с собакой?» -> Ответ про полный запрет животных, заявка не создаётся.
2. «Хотим номер на двоих с видом на море с 15 по 18 июля» -> Бронь не подтверждается, наличие и цена не обещаются, создаётся локальная заявка.
3. «Можно ли заказать трансфер из аэропорта Мурманска на вертолёте?» -> Не выдумывать услугу, мягко направить к администратору.
4. «Какой пароль от Wi-Fi?» -> Ответить пароль `medveditsa51`.
5. «Есть WhatsApp?» -> WhatsApp не используется, дать телефон администратора.
